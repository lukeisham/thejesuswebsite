Agent Architecture

Unified architecture reference for the AI Agent — covers the LLM backbone, configuration manifest, chat interaction, system monitoring, contact management, website mention tracking, and future widget expansion. Merges the former agent_backend_architecture and agent_chat_architecture docs.

  INTERACTION MODES (detected on frontend, applied on backend)
  ════════════════════════════════════════════════════════════

  detectInteractionMode(msg)          InteractionMode::from_str()
  ┌─────────────────────────┐         ┌──────────────────────────┐
  │ "run X", "do Y"         │ ──────> │ Execution                │
  │ → execution             │         │ Concise, no preamble     │
  ├─────────────────────────┤         ├──────────────────────────┤
  │ "what", "how", "why"    │ ──────> │ Collaborative (default)  │
  │ → collaborative         │         │ Iterative, 3 alternatives│
  ├─────────────────────────┤         ├──────────────────────────┤
  │ "check", "audit", "is"  │ ──────> │ Review                   │
  │ → review                │         │ Critical, security-first │
  ├─────────────────────────┤         ├──────────────────────────┤
  │ "high", "throttle"      │ ──────> │ Monitor                  │
  │ → monitor               │         │ Prompt, concise          │
  └─────────────────────────┘         └──────────────────────────┘

  HEAVY vs LIGHT ACTIONS
  ══════════════════════

  HEAVY (require verification)        LIGHT (execute immediately)
  ────────────────────────────        ─────────────────────────────
  "run wiki"   → run_wiki_sync       "run news"    → run_news_crawler
  "run challenge"→ run_challenge_     "run spelling"→ run_spelling_check
                   sort               "run scrape"  → run_page_scraper
  "run deadlink" → run_deadlinks
  "run embed"  → run_vector_embed

  Heavy → showVerificationPrompt()    Light → executeAgentAction()
  verification_required: true         executes immediately

  PROACTIVE ALERTING
  ══════════════════

  checkAlertThresholds(eventName, detail)
  fires on every widget event received

  ┌──────────┬─────────────────────────────────────────────┐
  │ Priority │ Alert Condition                              │
  ├──────────┼─────────────────────────────────────────────┤
  │ P1       │ Token usage > 80% (INFO/WARNING/CRITICAL)   │
  │ P2       │ Critical contacts > 0                       │
  │ P3       │ Draft backlog > 10                          │
  │ P5       │ No trace/reflection OR spelling errors > 20 │
  │ P6       │ Dead links > 0                              │
  │ P7       │ Security critical events > 5                │
  └──────────┴─────────────────────────────────────────────┘

  60-second deduplication per priority level
  30-second heartbeat (ping) when auto-monitoring enabled

Terms Glossary

Term Definition Source
DeepSeek API ⚠️ Intended LLM backbone for intent classification and natural-language responses. Not yet in codebase; current runtime uses CandleEngine (BERT). agent_guide.yml (planned)
CandleEngine Current BERT-based embedding engine (Candle ML, Metal/CPU). Produces 384-dim mean-pooled vectors. Stored as Option<Arc<CandleEngine>> on AppState — currently None at runtime. app_brain/src/candle.rs
agent_guide.yml YAML manifest (v1.2.0) that defines agent capabilities, widget list, logic constraints, P1–P8 priority hierarchy, and 4 interaction modes. Acts as the single source of truth for agent behavior. agent_guide.yml
Intent Rust enum with 8 variants (Essay, Wikipedia, Record, Challenge, Response, WidgetCommand, StatusReport, Unknown) classifying what a user message is asking the agent to do. app_brain/src/agent.rs
InteractionMode Rust enum with 4 variants (Execution, Collaborative, Review, Monitor) detected by frontend regex and forwarded as a field in the chat request. Applied as a vibe suffix on backend responses. app_brain/src/agent.rs
Heavy Action Widget command that requires user confirmation before execution (returns verification_required: true). Includes: run_wiki_sync, run_challenge_sort, run_deadlinks, run_vector_embed. wgt_agent-chat.js
Light Action Widget command that executes immediately without confirmation. Includes: run_news_crawler, run_spelling_check, run_page_scraper. wgt_agent-chat.js
Widget Event Bus Custom DOM event system used by dashboard widgets to broadcast state changes. The agent chat widget listens to 17 named events (e.g. TokenMetricsEvent, ContactTriageEvent) to build a real-time picture of system health. wgt_agent-chat.js
widgetState In-memory JS object on the agent chat widget that accumulates the latest payload from each widget event. Included as context in every chat request sent to the backend. wgt_agent-chat.js
Priority Hierarchy P1–P8 ordering defined in agent_guide.yml → current_focus: P1 Token, P2 Contact, P3 Drafts, P4 Research, P5 Reflection, P6 ChromaDB, P7 Metrics, P8 Server. Governs alert severity and status summary ordering. agent_guide.yml, agent.rs
Proactive Alerting Logic in checkAlertThresholds() that fires on every widget event and surfaces a contextual banner when a threshold is crossed (e.g. token usage >80%). Alerts deduplicate per priority key with a 60-second cooldown. wgt_agent-chat.js
Gatekeeper The sanitize() function in agent.rs that strips unsafe characters and truncates input to 16,384 characters before any LLM or classifier call. app_brain/src/agent.rs
try_widget_command() Function in agent.rs that maps a user message to one of 16 widget command types via keyword matching. Returns an AgentResponse with an optional verification_required flag for heavy actions. app_brain/src/agent.rs
build_status_summary() Function in agent.rs that queries token metrics, contacts, reflections, and work queue from SQLite and formats a P1–P8 status string returned in StatusReport intent responses. app_brain/src/agent.rs
Contact Triage Admin workflow where incoming contact form submissions (stored via store_contact()) are surfaced to the agent as P2 alerts. The admin marks them read via PATCH endpoint, removing them from the unread count. wgt_contact_triage.js, sqlite.rs
Mention Monitoring Workflow where wgt_page_metrics.js (P7) triggers a web scraper via POST /api/v1/tools/scraper/run to find external references to the site, persisting results via store_mention(). Monthly schedule is ⚠️ intended. wgt_page_metrics.js, sqlite.rs
Work Queue SQLite-backed queue of pending agent tasks retrieved by get_work_queue(). Surfaced in status summaries and visible in the Work Queue widget on the dashboard. app_storage/src/sqlite.rs
TraceReasoning Struct in thinking.rs that records the step-by-step logic path taken by the agent for a given request. Stored via get_trace_reasoning() and used for audit and reflection features. app_brain/src/thinking.rs
